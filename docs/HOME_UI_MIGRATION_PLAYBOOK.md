# 首页 UI 迁移经验总结（voice2text -> voice2text-flutter）

目标：把这次首页迁移里真正有效的方法、踩过的坑、以及已经验证过的修正路径沉淀下来，让后续从 RN 到 Flutter 的 UI 和组件迁移更稳定、更高保真、更可控。

---

## 1. 当前结论

这次首页迁移最重要的经验，不是“把 RN 组件翻译成 Flutter 组件”，而是分清三层合同：

- 视觉骨架合同：页面层次、间距、尺寸、圆角、浮层关系
- 状态合同：`loading / empty / normal / selection / recentlyDeleted`
- 数据与交互合同：`display_name / is_favorite / deleted_at_ms / group_name / folders / theme`

真正高质量的迁移，不是只把第一层做像，而是三层一起对齐。

---

## 2. 这次证明有效的方法

### 2.1 先保视觉骨架，再补复杂行为

首页一开始能快速接近 RN，是因为先钉住了这些值：

- 顶栏、分组栏、白色列表面板、左下 FAB 的层次关系
- 列表行高、内边距、标题层级、右侧动作区尺寸
- 空态在白色面板里的位置和占位密度

结论：

- 页面先“像”，后续补行为时才不容易把结构打散
- 视觉骨架的优先级高于一开始就追求完整功能

### 2.2 对齐的是布局关系，不是组件名

RN 里是：

- `SelectionModeLayout`
- `HomeTopActions`
- `GroupControlBar`
- `FileListRow`
- `ActionSheet`

Flutter 不需要逐个同名复刻，但必须对齐这些关系：

- 哪些块在同一层
- 哪些块参与滚动
- 哪些块是浮层
- 哪些块共享背景
- 哪些块是点击后弹层，不是页面内展开

经验：

- 迁移时优先画结构图，不要先写组件树

### 2.3 列表项和动作区要靠尺寸映射，不要靠感觉

这次首页还原度明显提升，主要因为列表项走了“尺寸映射”而不是“差不多”：

- 行高 `96`
- 内边距 `16`
- 右侧操作区 `36 x 36`
- 更多按钮 `18` 图标 + 固定点击盒子

经验：

- 这类基础密度必须直接对照 RN 源值
- 只要列表项密度偏了，整个页面就会立刻不像

### 2.4 先建 view model，再让 widget 渲染

首页后来变得可控，是因为展示层不再直接吃数据库字段，而是先收口到 `_RecordingPreview`。

这层至少负责：

- `title`
- `duration`
- `date`
- `favorite`
- `groupName`
- placeholder 与 real item 的差异

经验：

- UI 迁移不要让 widget 直接拼底层实体
- 否则格式化、占位策略、状态差异会散在多个组件里

### 2.5 页面 token 先局部收口，再考虑全局抽象

首页后面之所以能连续校细节，是因为先把一批值收到了 [lib/features/home/home_tokens.dart](/Users/studio/Documents/GitHub/voice2text-flutter/lib/features/home/home_tokens.dart:1)。

包括：

- 尺寸 token
- 字体层级 token
- action sheet token
- 顶部按钮点击盒子 token
- 亮暗模式 palette

经验：

- 页面先有自己的 token，才谈得上精修
- 没有 token 时，每次调整都在追散落常量

## 3. 这次踩过的坑

### 3.1 不要自己发明“迁移说明型占位”

这次前期最明显的偏差，是我补了一些并不存在于 RN 的解释性文案和占位行为，比如：

- “即将上线”
- “稍后接入”
- 额外的空态说明

这类内容对内部开发可能方便，但对高保真迁移是负资产。

结论：

- 没有 RN 基线的文案，不要发明
- 没有 RN 基线的组件样式，不要借别处通用组件硬套
- 功能没接完时，宁可静默，也不要写“开发中说明”

### 3.2 不能只迁骨架，不迁“状态级视觉合同”

早期空态出错，不是骨架错了，而是没有逐项对齐状态级合同：

- icon 错了
- icon 尺寸错了
- 空态宽度错了
- 空态文案语气错了

后来修正的方式是：

- 明确空态使用什么 icon
- 明确 icon 尺寸和线宽
- 明确空态容器要铺满列表面板宽度
- 明确不同 tab 的空态文案

经验：

- 还原页面不能只还原默认态
- 每个状态都是独立的视觉合同

### 3.3 不要把样式占位和真实数据状态混在一起

用户明确需要保留“全部音频”的样式占位，用来持续看页面密度。

后来证明正确做法是：

- 把 placeholder 明确做成单独分支
- 保留真实交互结构
- 但 placeholder 的副作用只在内存里发生

例如：

- placeholder 可重命名，但不写数据库
- placeholder 可收藏，但不写数据库
- placeholder 可删除，但只是从预览里移除

经验：

- “样式占位”不是“假装真实数据”
- 它应该是一个被明确隔离的 UI 预览态

### 3.4 高保真交互通常会逼出数据模型补齐

这次不是先设计数据库，而是 UI 一步步把底层缺口暴露出来了。

为了让首页交互真的可用，最后补了这些字段或表：

- `display_name`
- `is_favorite`
- `deleted_at_ms`
- `group_name`
- `folders`
- `is_dark_mode`

经验：

- 如果 RN 首页的交互依赖状态字段，Flutter 侧迟早要补齐
- 只迁 UI 不迁数据合同，最终一定会停在“看起来像，点起来不像”

### 3.5 主题切换不能只切 `ThemeMode`，页面 palette 也要跟着变

这次深色模式刚接的时候，一个明显风险是：

- `MaterialApp.themeMode` 切了
- 但首页颜色如果还是写死亮色，视觉上几乎不会变

后来的修正是：

- 让主题状态持久化
- 让首页使用动态 `HomePagePalette`
- 让页面文字、分割线、面板、点击态都跟随亮暗模式变化

经验：

- 主题迁移本质上是“语义色迁移”
- 不是单纯给 app 加一个 darkTheme

### 3.6 点击态不是“临时加背景”，而是“设计点击命中区”

顶部按钮的点击态一开始不对，根因不是颜色，而是点击区太小。

后来的稳定做法是：

- 固定点击盒子尺寸
- 默认透明
- 按下时在完整点击盒子里显示 overlay
- 用圆角裁切 overlay

经验：

- 点击态完整不完整，关键在 hit area，不在 icon 本身
- `padding` 不是唯一实现方式，但一定要有“比 icon 大的承载区”

## 4. 现在已经验证有效的迁移顺序

以后迁一个 RN 页面，建议固定按这个顺序：

1. 先抽页面结构树：顶栏、面板、列表、浮层、底部工具条。
2. 再抽状态树：normal、loading、empty、selection、deleted、error。
3. 再抽数据合同：哪些字段是 UI 必需字段，哪些是交互字段。
4. 先做静态高保真布局，只追“像”。
5. 把尺寸、颜色、文字层级收口到页面 token。
6. 用 view model 隔离 repository 和 widget。
7. 接只读数据和空态。
8. 再逐项补交互，优先补真实闭环，不补伪功能。
9. 最后补亮暗主题、批量操作、边缘状态。

这个顺序的重要性在于：

- 不会一上来就被业务逻辑拖乱结构
- 也不会停在只有静态图的阶段

## 5. 行为迁移的经验

### 5.1 action sheet 要迁“角色”，不要只迁“样子”

首页 action sheet 后来能开始像 RN，不是因为样式画得像，而是因为角色和语义开始对齐：

- 普通项：`重命名 / 移动到 / 收藏(取消收藏) / 分享 / 删除`
- 最近删除项：`恢复 / 彻底删除`

经验：

- action sheet 里的文案不是固定字符串
- 它取决于 item 当前状态和所在页面状态

### 5.2 删除链路必须区分软删除和硬删除

首页后来一个关键修正是：

- 普通列表 `删除` -> `移入最近删除`
- 最近删除 `删除` -> `彻底删除`

经验：

- 迁移行为时要先问“这个按钮在 RN 里改的是什么状态”
- 不要把所有删除都做成物理删除

### 5.3 详情打开链路优先复用，不要新造页面

Flutter 侧没有 RN 的 `/import-audio` 对应详情页时，正确做法不是造一个半成品路由，而是先复用现有详情能力，形成最小闭环。

经验：

- 功能不完整时，先复用已有可用路径
- 不要为了“结构看起来像”就新增未闭环的页面

## 6. 组件迁移经验

### 6.1 组件先迁“交互盒子”，再迁视觉细节

例如顶部按钮、更多按钮、选择框，都证明了一点：

- 先把尺寸、点击区、圆角、overlay 机制做对
- 再讨论图标、灰度、线宽

因为用户真正感受到的不是 icon 本身，而是整个交互盒子。

### 6.2 自定义弹层比临时 `ListTile` 更适合做高保真迁移

首页 action sheet 一开始如果用简单 `ListTile`，会立刻丢这些细节：

- 头部标题节奏
- 图标盒子宽度
- 选项间分割线
- 底部取消区独立节奏
- disabled opacity

经验：

- 高保真弹层优先自绘结构
- 通用组件只适合先做低保真可用版

### 6.3 页面内常用交互元件，优先统一成 token

这次已经比较明显需要统一的有：

- 顶部 icon 按钮盒子
- 列表 trailing 按钮盒子
- action sheet option 行高和 icon box
- selection toolbar button

经验：

- 这些如果不统一，页面虽然功能能跑通，但一眼会显得松散

## 7. 以后迁移 UI 时的判断标准

一个页面迁移完成，不应该只看“第一眼像不像”，而应该同时满足下面四条：

- 结构像：空间关系和信息层级稳定
- 状态像：empty / selection / deleted / action sheet 等都对齐
- 行为像：按钮改的是正确状态，不是假交互
- 可调像：颜色、尺寸、字重、点击态能通过 token 稳定调整

如果只满足前两条，那是高保真静态版。

如果四条都满足，才算真正迁移收口。

---

## 8. 针对后续 RN -> Flutter UI 迁移的执行清单

以后迁一个新页面，建议固定走这 10 步：

1. 从 RN 页面抽结构树、状态树、交互树。
2. 列出尺寸表、字体表、颜色语义表。
3. 明确哪些状态必须高保真，哪些状态可以先占位。
4. 先做静态骨架和列表密度。
5. 建页面 token。
6. 建 view model。
7. 接只读数据。
8. 逐项补 action、dialog、sheet、selection。
9. 补底层状态字段或表，不让 UI 假装可用。
10. 用截图、尺寸、文案、行为四类证据做 parity 收口。

---

## 9. 这次首页迁移留下的可复用原则

- 不要自由发挥文案和空态。
- 不要只看默认态，要看每个状态的视觉合同。
- 不要把样式占位和真实数据混成一套逻辑。
- 不要只迁 UI，不迁支撑交互的数据字段。
- 不要只切 themeMode，不迁页面 palette。
- 不要只改 icon，要先把点击盒子做对。
- 不要把“能点”当成“迁移完成”，要看点下去改的是不是 RN 里的同一状态。

最终判断标准：

- 不只是“像”
- 而是“继续改也不容易走样”
- 并且“样式、状态、行为都能解释得清楚”
